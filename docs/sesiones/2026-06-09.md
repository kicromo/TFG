# Sesión 2026-06-09 — Test de tolerancia a fallos

**Objetivo**: Simular una caída y recuperación de red, verificar que no se pierden segmentos, y medir los tiempos de detección y recuperación.

**Máquinas**: RPi5 (192.168.1.38) · PC (192.168.1.50)
**Herramienta**: `tests/test_fault_tolerance.py` (ejecución con sudo en RPi5)

---

## 1. Primer intento — resultado inválido

Se bajó solo la interfaz Ethernet:

```bash
sudo ip link set eth0 down
```

**Observado**: El RPi5 hizo failover automático a la interfaz WiFi (`wlan0`, 192.168.1.38). El sender se reconectó inmediatamente por WiFi. Solo se acumuló 1 segmento en `ready/`.

**Conclusión**: El test no simuló una caída real de conectividad. Es necesario desconectar todas las interfaces de red simultáneamente.

---

## 2. Problema adicional: SSH cae con la red

Al bajar ambas interfaces (`eth0` y `wlan0`), la sesión SSH desde el PC al RPi5 se interrumpe inmediatamente. Los procesos del capturer y del sender mueren.

**Solución**: Usar `tmux` en el RPi5. Los procesos iniciados dentro de una sesión tmux sobreviven a la desconexión del SSH y continúan ejecutándose.

```bash
# En RPi5, antes del test
tmux new-session -d -s camara
tmux send-keys -t camara "python3 /home/pi_damh/camera/nodo-remoto/capture/capturer.py" Enter
tmux send-keys -t camara " " ""
tmux new-window -t camara
tmux send-keys -t camara "python3 /home/pi_damh/camera/nodo-remoto/sync/sender.py" Enter
```

Tras reconectar el SSH después del test:

```bash
tmux attach -t camara
```

## 4. Análisis del tiempo de recuperación

Los 17.5 s de recuperación se deben al backoff exponencial del sender (secuencia: 5 s, 10 s, 30 s…). Cuando la red cayó:
- t = 1.5 s: primer intento de envío, falla, espera 5 s.
- t = 6.5 s: segundo intento, falla, espera 10 s.
- t = 16.5 s: tercer intento, falla, entra en espera de 30 s.
- t = 30 s: la red se restaura, pero el sender está esperando.
- t = 46.5 s: cuarto intento, éxito.

El tiempo de recuperación es 46.5 − 30 = **17.5 s** porque el sender estaba en medio de la espera de 30 s cuando volvió la red. En el peor caso (backoff máximo de 120 s), la recuperación podría tardar hasta 120 s.

**Conclusión**: El backoff exponencial es demasiado lento para reconexión, especialmente en el escenario de vigilancia en tiempo real. Se requiere un mecanismo de detección de reconexión proactivo.

## 5. Reboot inesperado del RPi5 durante las pruebas

Durante las pruebas se observó un reinicio no planificado del RPi5. Diagnóstico:

```bash
vcgencmd measure_temp     → 48.8°C  (dentro de límites normales)
free -h                   → 795 MB disponibles
vcgencmd get_throttled    → throttled=0x0  (sin throttling ni undervoltage)
```

No se pudo determinar la causa exacta porque el journal no era persistente. Se habilitó el journal persistente para futuras sesiones:

```bash
sudo mkdir -p /var/log/journal
sudo systemd-tmpfiles --create --prefix /var/log/journal
sudo systemctl restart systemd-journald
```
