## 1. Servicios systemd instalados en el RPi5

Antes de esta sesión, el capturer y el sender había que arrancarlos a mano. Si el RPi5 perdía la alimentación, la cámara dejaba de grabar indefinidamente hasta que alguien se conectara por SSH a reiniciarlos.

Se crearon tres archivos de servicio en `nodo-remoto/systemd/` y se instalaron en `/etc/systemd/system/` del RPi5:

| Servicio | Proceso | Usuario | Reinicio automático |
|----------|---------|---------|-------------------|
| `camera-capture.service` | `capturer.py` (FFmpeg) | root | Sí — 5 s tras fallo |
| `camera-sender.service` | `sender.py` (LIVE/BUFFER) | pi_damh | Sí — 10 s tras fallo |
| `camera-sync.service` | `sync_scheduler.py` | pi_damh | Sí — 30 s tras fallo |

Los tres servicios dependen de `network.target` y `wg-quick@wg0.service`, por lo que systemd los arranca en el orden correcto: primero la red, luego WireGuard, luego capturer y sender.

### Instalación

```bash
# Copiar service files al sistema
sudo cp /home/pi_damh/camera/nodo-remoto/systemd/*.service /etc/systemd/system/

# Registrar y habilitar
sudo systemctl daemon-reload
sudo systemctl enable camera-capture.service camera-sender.service camera-sync.service

# Arrancar inmediatamente
sudo systemctl start camera-capture.service camera-sender.service camera-sync.service
```

### Verificación de estado

```bash
sudo systemctl status camera-capture.service camera-sender.service camera-sync.service --no-pager
```

Resultado de la sesión:


## 3. Problema identificado: IP dinámica del PC (DHCP)

### Descripción

El router asigna IPs al PC de forma dinámica. En esta sesión el PC tuvo tres IPs distintas en el mismo día:

| Momento | IP cable (eno1) | Causa |
|---------|----------------|-------|
| Sesión anterior | 192.168.1.37 | Asignación previa |
| Primera conexión | 192.168.1.49 | Reinicio del router o lease expirado |
| Cable desconectado | Solo WiFi (192.168.1.47) | Cable físicamente desconectado |

Cada cambio de IP invalida el `Endpoint` del RPi5 en `/etc/wireguard/wg0.conf` y el túnel deja de funcionar.

### Solución a corto plazo: reserva DHCP

Configurar en el router que la MAC del cable del PC (`e0:70:ea:c9:c1:2b`) reciba siempre la misma IP. Procedimiento:

1. Abrir el panel del router: `http://192.168.1.1`
2. Buscar "DHCP" → "Reservas estáticas" o "Address Reservation"
3. Añadir: MAC `e0:70:ea:c9:c1:2b` → IP `192.168.1.50` (o cualquier IP fuera del pool dinámico)
4. Actualizar el Endpoint en el RPi5: `sudo sed -i 's/.*/Endpoint = 192.168.1.50:51820/' /etc/wireguard/wg0.conf`

Con esto el endpoint del RPi5 nunca queda obsoleto en la LAN.
