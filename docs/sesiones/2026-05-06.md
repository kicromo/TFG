# Sesión 2026-05-06 — Pruebas de codificación de vídeo

**Objetivo**: Instalar FFmpeg, probar el encoder hardware H.264 del RPi5 y validar la codificación software como alternativa. Confirmar los parámetros definitivos de captura.

**Máquina**: RPi5 (192.168.1.38)

## 2. Prueba del encoder hardware h264_v4l2m2m

Se esperaba que el RPi5 expusiera un encoder H.264 por V4L2 M2M, como hace el RPi4.

**Comando ejecutado**:

```bash
ffmpeg -f v4l2 -input_format mjpeg -video_size 1280x720 -framerate 15 \
  -i /dev/video0 \
  -c:v h264_v4l2m2m -b:v 1000k -t 5 /tmp/test.ts
```

**Resultado**:

```
Stream #0:0 -> #0:0 (mjpeg (native) -> h264 (h264_v4l2m2m))
[h264_v4l2m2m] Could not find a valid device
[h264_v4l2m2m] can't configure encoder
Conversion failed!
```

**Diagnóstico**: El RPi5 usa una arquitectura de procesamiento de imagen diferente al RPi4 (PiSP Backend, nodos `/dev/video20`–`/dev/video35`). Estos son nodos del ISP de imagen, no un encoder H.264 accesible mediante el framework V4L2 M2M. No hay encoder H.264 hardware disponible para FFmpeg en esta plataforma.

*Actualizado en DT-003.*

## 3. Prueba del encoder software libx264

```bash
ffmpeg -f v4l2 -input_format mjpeg -video_size 1280x720 -framerate 15 \
  -i /dev/video0 \
  -c:v libx264 -preset ultrafast -tune zerolatency -b:v 1000k \
  -t 5 /tmp/test.ts && echo "OK — $(du -h /tmp/test.ts | cut -f1)"
```

**Resultado**:

```
frame=74  fps=18  q=32.0  Lsize=635KiB  time=00:00:05.00  bitrate=1040.2kbits/s  speed=1.23x
[libx264] using cpu capabilities: ARMv8 NEON
[libx264] profile High 4:2:2, level 3.1
OK — 640K
```

**Uso de CPU durante la prueba** (monitoreado con `top`):

```
%Cpu(s):  6.1 us,  0.1 sy
FFmpeg:   ~22% de un núcleo (de 4 disponibles)
```

**Análisis de rendimiento**:

| Métrica | Valor obtenido | Objetivo | Resultado |
|---------|---------------|----------|-----------|
| FPS reales | 14.8 fps | 15 fps | Correcto |
| Speed | 1.23x | Mayor que 1.0x | Correcto (23% de margen) |
| Bitrate real | 991 kb/s | 1000 kb/s | Correcto |
| Tamaño 5 s | 640 KB | ~640 KB estimado | Correcto |
| CPU total | 6% | Menor del 50% | Correcto (margen amplio) |

**Conclusión**: `libx264 -preset ultrafast` es completamente viable en RPi5. El 6% de CPU total deja margen suficiente para ejecutar el capturer, el sender y el scheduler de forma simultánea.
