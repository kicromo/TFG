# Sesión 2026-07-16 — DuckDNS y port forwarding para IP pública dinámica (Fase 1.5)

**Objetivo**: Resolver el problema de la IP pública dinámica del PC antes de que el RPi5 viaje a Bolivia. La solución debe ser gratuita, funcionar con Movistar fibra y no requerir un VPS.

**Máquinas**:
- PC (nodo central): `192.168.1.49` (cable `eno1`), `10.0.0.1` (WireGuard), IP pública `79.147.126.98` (dinámica)
- RPi5 (nodo remoto): `192.168.1.38`, `10.0.0.2` (WireGuard)

---

## 1. Problema: IP pública dinámica del PC

El PC recibe la IP pública de Movistar de forma dinámica. No hay CGNAT — la IP es real y enrutable desde internet — pero puede cambiar en cualquier momento (reinicio del router, renovación de lease del ISP).

Cuando el RPi5 esté en Bolivia, necesitará conectar al endpoint WireGuard del PC. Si la IP cambia, el túnel deja de funcionar y no hay nadie en España que pueda corregirlo manualmente.

**Verificación de CGNAT**: un router con CGNAT tiene una IP privada en la WAN (100.x.x.x o 10.x.x.x). El router Askey RTF8225VW mostraba:

```
IP pública (WAN) : 79.147.126.98   <- IP real, enrutable desde internet
IP local (cable) : 192.168.1.49/24
```

Movistar fibra no usa CGNAT. El port forwarding es viable.

## 3. DuckDNS: DNS dinámico gratuito

### Qué es

DuckDNS es un servicio gratuito de DNS dinámico. Proporciona un subdominio fijo (`<nombre>.duckdns.org`) que siempre apunta a la IP pública actual del equipo que lo actualiza.

Cuando el ISP cambia la IP pública del PC, un script notifica a DuckDNS mediante una llamada HTTP a su API. DuckDNS actualiza el registro DNS del subdominio con la nueva IP. El RPi5 resuelve `rpidamh.duckdns.org` y obtiene siempre la IP correcta, sin necesidad de intervención manual.

### Subdominio creado

`rpidamh.duckdns.org` — apunta a la IP pública actual del PC.

### Script de actualización

Ruta en el PC: `/home/damh/.duckdns/update.sh`

```bash
#!/bin/bash
echo url="https://www.duckdns.org/update?domains=rpidamh&token=<TOKEN>&ip=" \
  | curl -k -o /home/damh/.duckdns/duck.log -K -
```

El script llama a la API de DuckDNS sin especificar IP; DuckDNS detecta automáticamente la IP pública del solicitante. La respuesta `OK` confirma que el registro DNS se actualizó correctamente.

**Verificación manual**:

```
$ bash /home/damh/.duckdns/update.sh
OK
```

### Cron configurado en el PC

```
*/5 * * * * /home/damh/.duckdns/update.sh
```

Cada 5 minutos el script se ejecuta. Si la IP no ha cambiado, DuckDNS responde `OK` sin modificar nada. Si cambió, actualiza el registro.

## 4. Port forwarding en el router

El router Movistar Askey RTF8225VW tiene un panel de administración accesible en `http://192.168.1.1`. Se configuró la siguiente regla:

| Campo | Valor |
|-------|-------|
| Protocolo | UDP |
| Puerto externo (WAN) | 51820 |
| Puerto interno (LAN) | 51820 |
| IP interna (destino) | 192.168.1.49 |

Con esta regla, cualquier paquete UDP que llegue a la IP pública del PC en el puerto 51820 se reenvía automáticamente al PC dentro de la LAN.

**Nota**: la IP interna `192.168.1.49` debería estar fija mediante una reserva DHCP (pendiente, ver sección 7). Sin esa reserva, si el router reasigna la IP del PC, el port forwarding apuntaría a la dirección equivocada.

---

## 5. Endpoint del RPi5 actualizado

Se modificó `/etc/wireguard/wg0.conf` en el RPi5:

```
# Antes
Endpoint = 192.168.1.49:51820

# Ahora
Endpoint = rpidamh.duckdns.org:51820
```

Con el hostname en lugar de una IP, WireGuard resuelve el nombre DNS cada vez que necesita establecer o reestablecer el túnel. Si la IP pública del PC cambia, el próximo intento de handshake resuelve el nombre y encuentra la nueva dirección.

Pasos ejecutados en el RPi5:

```bash
sudo wg-quick down wg0
sudo nano /etc/wireguard/wg0.conf   # editar Endpoint
sudo wg-quick up wg0
```

## 7. Consideraciones de seguridad

DuckDNS solo conoce la IP pública del PC — no intercepta, no ve, no almacena el tráfico de vídeo ni las claves WireGuard. Su función es equivalente a publicar la dirección postal de un edificio: permite encontrar el edificio, pero no da acceso al interior.

La puerta del edificio sigue siendo WireGuard con autenticación Curve25519. Solo el RPi5, que tiene la clave privada correspondiente a la clave pública registrada en el PC, puede participar en el túnel. Un atacante que conozca `rpidamh.duckdns.org` puede ver la IP pública del PC y enviar paquetes UDP al puerto 51820, pero WireGuard los descartará silenciosamente si no van firmados con la clave privada correcta.

| # | Incidencia | Causa | Diagnóstico | Solución |

---

### Latencia esperada por entorno de red

| Entorno | Latencia estimada | Motivo |
|---------|-------------------|--------|
| España — mismo operador (Movistar) | ~0.4 ms | Tráfico intra-carrier; no sale de la red del ISP |
| España — operadores distintos | 5–20 ms | Tráfico pasa por puntos de intercambio nacionales |
| Internacional España ↔ Bolivia | 200–300 ms | Trayecto intercontinental; pendiente de medir |

**Impacto en el pipeline**: los segmentos de vídeo tienen 3 segundos de duración. Una latencia de 300 ms representa el 10% de un segmento y no afecta a la entrega. El pipeline usa transferencia por acumulación, no streaming en tiempo real, por lo que los retardos de red son absorbidos sin pérdida de datos.
