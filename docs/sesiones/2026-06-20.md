## 1. sync_scheduler.py — envío nocturno del archive

**Fichero creado**: `nodo-remoto/sync/sync_scheduler.py`  
**Copiado a RPi5**: `/home/pi_damh/camera/nodo-remoto/sync/sync_scheduler.py`

### Responsabilidad

Proceso independiente del sender. Se ejecuta en bucle continuo y, a la hora programada (medianoche por defecto), envía al nodo central todos los segmentos acumulados en `archive/` durante caídas de red.

### Flujo de ejecución

```
Al arrancar:
  - Si quedan archivos en syncing/ (de una sync interrumpida anteriormente),
    se devuelven a archive/ antes de continuar.

Cada noche a la hora programada:
  1. Mueve archive/*.ts → syncing/  (operación atómica, evita conflictos con sender)
  2. Abre conexión TCP al receiver
  3. Por cada segmento:
       ACK  → borra de syncing/
       NACK → devuelve a archive/ (reintento la noche siguiente)
  4. Error de red → devuelve todos los pendientes a archive/
  5. Registra resumen: "enviados: 47/50 | fallidos: 3 | 18.2 MB transferidos"
  6. Calcula tiempo hasta la próxima ejecución y duerme
```

### Por qué un directorio temporal syncing/

El sender en modo BUFFER escribe segmentos en `archive/` continuamente. Si el scheduler nocturno leyera directamente de `archive/`, podría haber condiciones de carrera. Al mover el lote completo a `syncing/` antes de abrir la conexión TCP, el sender sigue escribiendo en `archive/` sin interferencia.

### Protocolo de envío

Mismo protocolo que `sender.py` (cabecera JSON con filename, size, sha256 + ACK + datos raw + confirmación). El receiver no distingue si el emisor es el sender en vivo o el scheduler nocturno.

### Parámetros configurables (config.yaml)

```yaml
sync:
  archive_dir: /home/pi_damh/buffer
  scheduled_hour: 0
  scheduled_minute: 0
```

### Test de arranque

```

[INFO] Próxima sync: 2026-06-21 00:00 (en 89 min)
```

## 2. event_logger.py — registro de eventos del nodo central

**Fichero creado**: `nodo-central/receiver/event_logger.py`

### Formato

JSON Lines (.jsonl): un objeto JSON por línea con timestamp ISO 8601. Legible por humanos y procesable con herramientas estándar (grep, jq, Python).

### Tipos de evento

| Evento | Descripción | Campos |
|--------|-------------|--------|
| connection_open | Nueva conexión TCP entrante | client |
| connection_close | Conexión cerrada con resumen | client, segments_ok, segments_error |
| segment_ok | Segmento verificado y almacenado | client, file, size_kb |
| hash_error | SHA-256 no coincide (evento de seguridad) | client, file |
| header_error | Cabecera JSON inválida | client, reason |

### Ejemplos de líneas en events.jsonl

```json
{"ts":"2026-06-20T22:05:01.123","event":"connection_open","client":"192.168.1.38:54321"}
{"ts":"2026-06-20T22:05:01.456","event":"segment_ok","client":"192.168.1.38","file":"seg_20260620_220501.ts","size_kb":384.0}
{"ts":"2026-06-20T22:05:02.100","event":"connection_close","client":"192.168.1.38:54321","segments_ok":5,"segments_error":0}
```

### Rotación automática

RotatingFileHandler de Python: rotación al superar 10 MB, conservando 7 copias (maximo ~70 MB en disco).

### Verificación de arranque

```
2026-06-20 22:09:54 [INFO] Receiver escuchando en 0.0.0.0:9000
→ Crea automáticamente: nodo-central/logs/events.jsonl
```

### Consultas útiles

```bash
tail -f nodo-central/logs/events.jsonl

grep '"hash_error"' nodo-central/logs/events.jsonl

grep '"segment_ok"' nodo-central/logs/events.jsonl | grep "$(date +%Y-%m-%d)" | wc -l
```
