# Sesión 2026-07-05 — Despliegue WireGuard VPN (Fase 1)

**Objetivo**: Establecer el túnel WireGuard entre el PC (nodo central) y el RPi5 (nodo remoto) como primer paso hacia el despliegue en redes distintas.

**Máquinas**:
- PC (nodo central): `192.168.1.37` (cable `eno1`), `192.168.1.47` (WiFi `wlo1`)
- RPi5 (nodo remoto): `192.168.1.38`

---

## 1. Generación de claves y configuración

WireGuard autentica mediante pares de claves asimétricas (Curve25519). Cada extremo genera su propio par; solo se intercambia la clave pública.

```bash
# Generar par de claves (ejecutado en cada máquina)
wg genkey | tee private.key | wg pubkey > public.key
```

### Config PC (`/etc/wireguard/wg0.conf`)

```ini
[Interface]
PrivateKey = <clave privada PC>
Address = 10.0.0.1/24
ListenPort = 51820

[Peer]
# RPi5 — nodo remoto
PublicKey = <clave publica RPi5>
AllowedIPs = 10.0.0.2/32
```

El PC actua como servidor: tiene `ListenPort` fijo y no necesita `Endpoint` porque aprende la IP del RPi5 dinámicamente al recibir el primer paquete.

### Config RPi5 (`/etc/wireguard/wg0.conf`)

```ini
[Interface]
PrivateKey = <clave privada RPi5>
Address = 10.0.0.2/24

[Peer]
# PC — nodo central
PublicKey = <clave publica PC>
AllowedIPs = 10.0.0.1/32
Endpoint = 192.168.1.37:51820
PersistentKeepalive = 25
```

El RPi5 actua como cliente: tiene `Endpoint` explícito (donde escucha el servidor) y `PersistentKeepalive = 25` para mantener el túnel activo a través de NATs y recuperarse de interrupciones.

### Firewall en el PC

```bash
sudo ufw allow 51820/udp
```

## 2. Incidente durante el despliegue: endpoint con IP incorrecta

### Síntoma

Después de ejecutar `wg-quick up wg0` en ambas máquinas, el RPi5 enviaba paquetes pero el PC no recibía ninguno. El comando `sudo wg show` en el RPi5 mostraba:

```
transfer: 5.35 KiB sent, 0 received
```

El PC no registraba handshake ni transferencia.

### Diagnóstico

```bash
# En el PC — Terminal 1: capturar UDP en la interfaz física
sudo tcpdump -i eno1 -n udp port 51820

# En el RPi5 — Terminal 2: forzar tráfico
ping -c 5 10.0.0.1
```

El tcpdump no capturó ningún paquete. Los paquetes no llegaban al PC.

```bash
# Verificar IP real del PC
ip addr show | grep '192.168'
```

Resultado:
```
inet 192.168.1.37/24 brd 192.168.1.255 scope global dynamic noprefixroute eno1
inet 192.168.1.47/24 brd 192.168.1.255 scope global dynamic noprefixroute wlo1
```

**Causa raíz**: el config del RPi5 tenía `Endpoint = 192.168.1.50:51820`. Esa IP no pertenecía al PC (probablemente era una IP asumida de sesiones anteriores o de otra máquina en la red). Los paquetes WireGuard iban a un destino incorrecto y el PC nunca los veía.

### Resolución

```bash
# En el RPi5
sudo wg-quick down wg0
sudo sed -i 's/192.168.1.50/192.168.1.37/' /etc/wireguard/wg0.conf
sudo wg-quick up wg0
```

```bash
# En el PC — reiniciar para limpiar estado
sudo wg-quick down wg0 && sudo wg-quick up wg0
```

El `PersistentKeepalive = 25` del RPi5 envió un keepalive al nuevo endpoint correcto en los siguientes 25 segundos. El handshake se completó automáticamente.

### Verificación del túnel

```bash
# Desde el PC
ping -c 3 10.0.0.2
```
```
64 bytes from 10.0.0.2: icmp_seq=1 ttl=64 time=0.441 ms
64 bytes from 10.0.0.2: icmp_seq=2 ttl=64 time=0.451 ms
64 bytes from 10.0.0.2: icmp_seq=3 ttl=64 time=0.462 ms
0% packet loss
```

```bash
# Desde el RPi5
ping -c 3 10.0.0.1
```
```
64 bytes from 10.0.0.1: icmp_seq=1 ttl=64 time=0.244 ms
64 bytes from 10.0.0.1: icmp_seq=2 ttl=64 time=0.843 ms
64 bytes from 10.0.0.1: icmp_seq=3 ttl=64 time=0.402 ms

```

| Dirección | Latencia media | Pérdida |
|-----------|---------------|---------|
| PC → RPi5 (10.0.0.2) | 0.45 ms | 0% |
| RPi5 → PC (10.0.0.1) | 0.50 ms | 0% |

Latencia esperada en la misma LAN. Cuando la RPi5 esté en Bolivia, será de 200–300 ms, lo que sigue siendo compatible con el protocolo TCP de segmentos.

## 3. Escenarios de fallo y cómo prevenirlos

### 3.1 El túnel no levanta al reiniciar

**Causa**: WireGuard no está habilitado como servicio del sistema.

**Síntoma**: después de reiniciar cualquiera de las máquinas, `ping 10.0.0.2` o `ping 10.0.0.1` falla. `ip addr show wg0` da error (`Device not found`).

**Prevención**:
```bash
# En ambas máquinas
sudo systemctl enable wg-quick@wg0
```

**Diagnóstico si ocurre**: `sudo systemctl status wg-quick@wg0` para ver si el servicio está inactivo.

---

### 3.2 La IP del PC cambia (DHCP)

**Causa**: el PC obtiene una IP diferente del router en el próximo arranque (DHCP sin reserva).

**Síntoma**: el RPi5 intenta conectar a 192.168.1.37 pero esa IP ya no es del PC. El tcpdump en el PC no muestra paquetes. `wg show` en RPi5 muestra `0 received`.

**Prevención**:
- Configurar reserva DHCP en el router para la MAC del PC (dirección estática por DHCP).
- O en Fase 2, migrar a VPS con IP pública fija, que resuelve este problema permanentemente.

**Diagnóstico si ocurre**:
```bash
# Verificar IP actual del PC
ip addr show | grep '192.168'
# Si cambió, actualizar en RPi5
sudo sed -i 's/<ip_antigua>/<ip_nueva>/' /etc/wireguard/wg0.conf
sudo wg-quick down wg0 && sudo wg-quick up wg0
```

---

### 3.3 El firewall bloquea el puerto 51820

**Causa**: una actualización del sistema, un reset de UFW, o un cambio de red puede revocar la regla de firewall.

**Síntoma**: tcpdump en el PC no muestra paquetes llegando (igual que el incidente de hoy pero sin el problema del Endpoint).

**Diagnóstico**:
```bash
sudo ufw status | grep 51820
```

**Corrección**:
```bash
sudo ufw allow 51820/udp
sudo ufw reload
```

---

### 3.4 El pipeline sigue usando la IP LAN en lugar de la IP WireGuard

**Causa**: después de establecer el túnel, el `config.yaml` del RPi5 todavía apunta a `192.168.1.37` (IP física) en lugar de `10.0.0.1` (IP del túnel).

**Síntoma**: el vídeo se transmite correctamente pero sin cifrado WireGuard — el tráfico TCP va directo por la LAN, no por el túnel.

**Comprobación**:
```bash
grep 'host:' /home/pi_damh/camera/nodo-remoto/config/config.yaml
```

**Corrección**:
```bash
sed -i 's/192.168.1.37/10.0.0.1/' /home/pi_damh/camera/nodo-remoto/config/config.yaml
```

Esto es el cambio de configuración que activa formalmente la Fase 1: el mismo código del sender, apuntando ahora al túnel cifrado.

## 4. Consideraciones de seguridad de WireGuard

### 4.1 Protección de claves privadas

Las claves privadas son la identidad de cada nodo. Si una clave privada se filtra, un atacante puede suplantar ese nodo completamente.

| Riesgo | Consecuencia | Mitigación |
|--------|-------------|------------|
| Clave privada en git | Cualquiera con acceso al repositorio puede suplantar el nodo | `.gitignore` incluye `wg*.conf` y `*.key` — verificado |
| Clave privada legible por otros usuarios | Usuario local puede leer la clave | `chmod 600 /etc/wireguard/wg0.conf` (wg-quick lo aplica automáticamente) |
| Backup de la clave en texto plano | Exposición al hacer backup del sistema | No incluir `/etc/wireguard/` en backups no cifrados |

Para verificar permisos:
```bash
ls -la /etc/wireguard/wg0.conf
# debe ser: -rw------- root root
```

### 4.2 Rotación de claves

Las claves actuales son válidas indefinidamente (WireGuard no tiene expiración nativa). Para renovarlas:

```bash
# En la máquina a rotar
wg genkey | sudo tee /etc/wireguard/private.key | wg pubkey > /tmp/new_public.key
# Actualizar PrivateKey en /etc/wireguard/wg0.conf
# Enviar new_public.key al otro extremo para actualizar su [Peer] PublicKey
# Reiniciar wg0 en ambas máquinas
```

Recomendación: rotar claves al menos una vez al año, o inmediatamente si se sospecha compromiso.

### 4.3 AllowedIPs como control de acceso

La directiva `AllowedIPs` actúa como lista blanca: solo se aceptan paquetes cuya IP de origen coincide. El PC solo acepta paquetes WireGuard con IP de origen `10.0.0.2/32` (el RPi5), y viceversa.

Si se añadiera un tercer nodo en el futuro (otro RPi o el VPS), habría que añadir explícitamente su IP en `AllowedIPs`, lo que evita accesos no intencionados.

### 4.4 Lo que WireGuard NO protege

| Amenaza | Estado |
|---------|--------|
| Autenticación de la aplicación (quién manda segmentos) | Pendiente — JWT en Fase 3 |
| Acceso no autorizado a la interfaz web | Pendiente — bcrypt + JWT |
| Validez del contenido del segmento | Cubierto por SHA-256 en el receiver |
| Privacidad del tráfico en tránsito | Cubierto por WireGuard (ChaCha20-Poly1305) |

WireGuard garantiza que nadie en la red puede leer ni modificar los segmentos en tránsito. La identidad a nivel de aplicación (quién es el cliente que se conecta al receiver) se cubrirá con JWT en fases posteriores.

## 6. Problema recurrente: IP del PC cambia por DHCP

En cada sesión de trabajo el router asigna una IP diferente al PC:

| Sesión | IP cable (eno1) | IP WiFi (wlo1) |
|--------|----------------|----------------|
| 2026-07-05 (primera) | 192.168.1.37 | 192.168.1.47 |
| 2026-07-05 (segunda) | 192.168.1.49 | 192.168.1.47 |

Cada vez que la IP del PC cambia, el Endpoint del RPi5 queda obsoleto y el túnel no establece handshake. Síntoma característico: `wg show` en RPi5 muestra `latest handshake: X days ago` y el ping a 10.0.0.1 da 100% de pérdida.

### Procedimiento cuando el túnel no levanta

```bash
# 1. Verificar IP actual del PC
ip addr show | grep '192.168'

# 2. Actualizar endpoint en el RPi5 (sustituir <nueva_ip> por lo que devuelva el paso 1)
sudo wg-quick down wg0
sudo sed -i 's/192.168.1.XX/<nueva_ip>/' /etc/wireguard/wg0.conf
sudo wg-quick up wg0

# 3. Verificar túnel
ping -c 3 10.0.0.1   # desde el RPi5
```

### Solución permanente (pendiente)

Configurar reserva DHCP en el router para las MACs del PC:
- Cable (`eno1`): `e0:70:ea:c9:c1:2b`
- WiFi (`wlo1`): `c8:94:02:aa:81:0b`

Asignarle siempre la misma IP (por ejemplo `192.168.1.50`) elimina el problema hasta la Fase 2, donde el endpoint será un VPS con IP pública fija.

## 7. Flujo completo de datos con WireGuard activo

Este diagrama describe el camino de cada segmento de vídeo desde la cámara hasta el almacenamiento en el PC, incluyendo la capa WireGuard.

```
NODO REMOTO (RPi5 — Bolivia en el futuro)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[Cámara Logitech C270]
    │  USB · MJPG · 1280×720 · 15 fps
    ▼
[capturer.py — FFmpeg]
    │  Recodifica MJPG → H.264 (libx264 ultrafast)
    │  Genera segmentos MPEG-TS de 3 s exactos (GOP = 45 frames)
    │  Escribe en staging/ → mueve atómicamente a ready/
    ▼
[sender.py — modo LIVE]
    │  Lee segmento de ready/
    │  Calcula SHA-256 del contenido
    │  Abre TCP hacia 10.0.0.1:9000     ← IP del túnel WireGuard
    │  Envía: JSON header + espera ACK + datos binarios + espera confirmación
    │  Si ACK: borra segmento de ready/
    │  Si NACK o fallo: mueve a archive/ (enviará el scheduler nocturno)
    ▼
[WireGuard — kernel RPi5]
    │  El paquete TCP destino 10.0.0.1 entra en la interfaz wg0
    │  WireGuard cifra con ChaCha20-Poly1305
    │  Encapsula en UDP y envía a 192.168.1.49:51820   ← IP física del PC

━━━━━━━━━━━━━━━━━━━━━━
        RED LOCAL (LAN) / Internet en Fase 2
        UDP cifrado · puerto 51820
━━━━━━━━━━━━━━━━━━━━━━

NODO CENTRAL (PC — España)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[WireGuard — kernel PC]
    │  Recibe UDP en puerto 51820 (eno1 o wlo1)
    │  Verifica que el paquete viene de la clave pública del RPi5
    │  Descifra y entrega TCP a la interfaz wg0 (10.0.0.1)
    ▼
[receiver.py — escucha en 0.0.0.0:9000]
    │  Acepta conexión TCP del sender
    │  Lee JSON header: { filename, size, sha256 }
    │  Envía ACK de cabecera
    │  Lee datos binarios (exactamente size bytes)
    │  Calcula SHA-256 de los datos recibidos
    │  Compara con sha256 de la cabecera
    │    ├─ Coincide  → escribe en storage/video/YYYY-MM-DD/HH/  → ACK
    │    └─ No coincide → descarta segmento → NACK → hash_error en log
    ▼
[event_logger.py]
    │  Registra en logs/events.jsonl:
    │    connection_open, segment_ok / hash_error, connection_close
    ▼
[Almacenamiento]
    nodo-central/storage/video/
    └── 2026-07-05/
        └── 14/
            ├── 2026-07-05_14-00-00.ts
            ├── 2026-07-05_14-00-03.ts
            └── ...
```

### Qué aporta WireGuard en este flujo

| Sin WireGuard (Fase 0) | Con WireGuard (Fase 1+) |
|------------------------|------------------------|
| TCP en claro por la LAN | TCP cifrado dentro del túnel UDP |
| Cualquier máquina de la LAN puede interceptar el tráfico | Solo el par con la clave privada correcta puede descifrar |
| El sender apunta a la IP física del PC (cambia con DHCP) | El sender apunta a 10.0.0.1 (IP fija del túnel, no cambia nunca) |
| Sin autenticación de red | Solo el RPi5 con su clave privada puede ser peer del PC |

El cambio en `config.yaml` es lo único que diferencia Fase 0 de Fase 1:
```yaml
# Fase 0
server:
  host: "192.168.1.49"   # IP física, sin cifrado

# Fase 1 (actual)
server:
  host: "10.0.0.1"       # IP del túnel WireGuard, cifrado transparente
```
