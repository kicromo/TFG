# Sesión 2026-05-27 — Sender, receiver y primera transmisión LAN

**Objetivo**: Implementar el módulo de envío en el RPi5 y el receptor en el PC, y verificar la primera transmisión LAN extremo a extremo.

**Máquinas**: RPi5 (192.168.1.38) como nodo remoto · PC (192.168.1.50) como nodo central

---

## 1. Diseño del protocolo de transferencia

Se define el protocolo por segmento (JSON sobre TCP, sin biblioteca externa):

```
RPi5 → PC  :  cabecera JSON + '\n'    { "filename": "seg_XXX.ts", "size": 393216, "sha256": "abc..." }
PC   → RPi5:  respuesta JSON + '\n'   { "status": "ok" }                  ← ACK de cabecera
RPi5 → PC  :  [size bytes de datos raw del segmento]
PC   → RPi5:  confirmación JSON + '\n' { "status": "ok" }                  ← confirmación de almacenamiento
```

Si la verificación SHA-256 falla en el PC, el receiver responde `{ "status": "error", "reason": "hash_mismatch" }` y el sender archiva el segmento para reintento.

---

## 2. Implementación del sender.py (primera versión)

**Fichero**: `nodo-remoto/sync/sender.py`

Funcionalidades implementadas:
- Vigila el directorio `ready/` y envía los segmentos en orden cronológico.
- Calcula SHA-256 del segmento antes del envío.
- Envía cabecera JSON, espera ACK, envía datos, espera confirmación.
- Si la conexión falla, registra el error y reintenta con backoff exponencial.
- Al recibir confirmación del receiver, elimina el segmento de `ready/`.

---

## 3. Implementación del receiver.py

**Fichero**: `nodo-central/receiver/receiver.py`

Funcionalidades implementadas:
- Servidor TCP multihilo en `0.0.0.0:9000` (ThreadedTCPServer).
- Por cada segmento: lee cabecera JSON, responde ACK, recibe datos, verifica SHA-256.
- Si el hash coincide: almacena en `storage/video/YYYY-MM-DD/HH/seg_XXX.ts` y confirma.
- Si el hash no coincide: registra warning y envía NACK.

**Estructura de almacenamiento**:

```
storage/video/
└── 2026-05-27/
    └── 10/
        ├── seg_20260527_100012.ts
        ├── seg_20260527_100015.ts
        └── seg_20260527_100018.ts
```

## 4. Primera transmisión LAN — obstáculo con UFW

Al intentar conectar el sender del RPi5 al receiver del PC, las conexiones expiraban. El PC tiene UFW activo.

**Intento inicial** (fallo):

```bash
sudo ufw allow from 192.168.1.38 to any port 9000/tcp
```

```
ERROR: Puerto erróneo '9000/tcp'
```

**Corrección de sintaxis**:

```bash
sudo ufw allow from 192.168.1.38 to any port 9000 proto tcp
```

La regla se añadió correctamente, pero las conexiones seguían expirando.

**Diagnóstico**: El PC tiene dos interfaces de red activas simultáneamente:
- `eno1`: 192.168.1.50 (Ethernet)
- `wlo1`: 192.168.1.42 (WiFi)

La regla UFW con `from 192.168.1.38` no estaba siendo evaluada correctamente por la asimetría de rutas entre interfaces. Solución temporal para el entorno de desarrollo:

```bash
sudo ufw allow 9000/tcp
```

Pendiente de revisar la regla más específica en el entorno de producción.

## 5. Primera transmisión LAN completa

Con el receiver activo en el PC y el capturer en el RPi5:

```bash
# PC
python3 receiver.py

# RPi5
python3 sender.py
```

**Salida del sender**:

```
[INFO] Conectado a 192.168.1.50:9000 — 1 segmento(s) pendiente(s)
[INFO] Enviando seg_20260527_100012.ts (384 KB)…
[INFO] Confirmado y eliminado: seg_20260527_100012.ts
[INFO] Enviando seg_20260527_100015.ts (382 KB)…
```

**Resultado en el PC tras 30 minutos**:

```

```

**Latencia extremo a extremo** (captura hasta almacenamiento en PC): 3 a 6 segundos (duración del segmento más transmisión TCP a 0.16 ms RTT).
