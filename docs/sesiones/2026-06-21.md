## 1. Script de concatenación: concat_to_mp4.sh

**Fichero creado**: `scripts/concat_to_mp4.sh`

Concatena los segmentos .ts de una fecha y hora determinadas en un único archivo .mp4, sin re-encodear (copia de streams, rápido y sin pérdida de calidad).

Utiliza el demuxer `concat` de FFmpeg con una lista de archivos ordenada cronológicamente.

### Uso

```bash
./scripts/concat_to_mp4.sh YYYY-MM-DD [HH] [output.mp4]
```

**Ejemplos**:

```bash
./scripts/concat_to_mp4.sh 2026-06-20           # todo el dia → 2026-06-20.mp4
./scripts/concat_to_mp4.sh 2026-06-20 14        # solo hora 14 → 2026-06-20_14.mp4
./scripts/concat_to_mp4.sh 2026-06-20 15 clip.mp4  # nombre personalizado
```

### Corrección aplicada durante el desarrollo

`dirname "$0"` producía una ruta relativa cuando el script se ejecutaba desde otro directorio. Los paths del fichero de concat resultaban relativos al directorio temporal `/tmp`, donde FFmpeg intentaba resolverlos. Fix:

```bash
STORAGE_DIR="$(realpath "$(dirname "$0")/../nodo-central/storage/video")"
```

## 4. Test de integridad SHA-256

**Fichero creado**: `tests/test_sha256_integrity.py`

### Objetivo

Verificar que el receiver detecta y rechaza correctamente segmentos con datos manipulados, y que el logger de eventos registra cada incidente como auditoría de seguridad.

### Escenarios probados

| Escenario | Descripcion | Respuesta esperada | Resultado |
|-----------|-------------|-------------------|-----------|
| 1 | Segmento con datos y hash correctos | ACK | PASS |
| 2 | Hash falso en la cabecera (metadatos manipulados) | NACK — hash_mismatch | PASS |
| 3 | Datos manipulados en tránsito (bit-flip post-firma) | NACK — hash_mismatch | PASS |

Los tres escenarios se ejecutan en una única conexión TCP. Tras un NACK por hash_mismatch, el receiver continúa atendiendo la misma sesión (no cierra la conexión), lo que permite encadenar los escenarios sin reconectar.

### Cómo funciona el escenario 3

```
Sender calcula SHA-256 del segmento original → H_original
Atacante modifica bytes 512 y 1024 del segmento en tránsito
Receiver recibe:
  cabecera: { sha256: H_original }   ← hash del original (inalterado)
  datos:    [segmento modificado]    ← bytes diferentes
Receiver computa SHA-256 de los datos recibidos → H_recibido
H_recibido != H_original → rechaza → hash_error → NACK
```

### Salida del test

```
==============================================================

Receiver: 127.0.0.1:9000
==============================================================

Escenario                                     Esperado   Obtenido   Resultado
--------------------------------------------------------------
  Segmento con datos y hash correctos         ACK        ACK        PASS
  Hash falso en cabecera (metadatos manipulados) NACK    NACK       PASS
  Datos manipulados en tránsito (bit-flip)    NACK       NACK       PASS

RESULTADO FINAL: 3/3 escenarios correctos

hash_error registrados en events.jsonl: 2 (esperado: 2)
Correcto — los intentos de manipulacion quedan auditados.
```

Este test demuestra que el mecanismo de integridad SHA-256 (DT-005) funciona correctamente frente a dos vectores de ataque distintos:

| Vector de ataque | Detectado | Evidencia |
|-----------------|-----------|-----------|
| Modificación de metadatos (cabecera) | Si | hash_error en events.jsonl |
| Modificación de datos en tránsito (tampering) | Si | hash_error en events.jsonl |
